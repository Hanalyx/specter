## 2.1 Understanding Intent Drift

### What Is Intent Drift?

Intent drift occurs when AI-generated code gradually deviates from the original specification's intent while still producing functionally correct output. It's a subtle, insidious problem because:

1. **The code works.** It passes tests. It produces correct results.
2. **The code looks reasonable.** It follows good practices -- just not *your* specified practices.
3. **The drift is cumulative.** One small deviation leads to another, and before you know it, the codebase has become an inconsistent mess.
4. **The AI doesn't know it's drifting.** It doesn't have a persistent memory of your architectural decisions. Each prompt is a fresh start.

### The Three Types of Intent Drift

#### Type 1: Architectural Drift

The spec says "use technology X," but the AI uses technology Y.

```markdown
# Spec says:
"All state management SHALL use Zustand stores."

# AI generates:
import { useState, useEffect } from 'react';
// Uses React local state instead of Zustand
```

The component works fine. But you now have inconsistent state management across your codebase. Some components use Zustand (as specified), others use local state (as the AI defaulted to).

#### Type 2: Pattern Drift

The spec defines a coding pattern, but the AI uses a different (often equally valid) pattern.

```markdown
# Spec says:
"Error handling SHALL use Result types (Ok/Err pattern)."

# AI generates:
try {
  const data = await fetchUser(id);
  return data;
} catch (error) {
  throw new HttpException(error.message, 500);
}
// Uses try/catch instead of Result types
```

Both approaches handle errors. But the spec chose Result types for a reason -- perhaps for better composability, or to avoid thrown exceptions in the hot path. The AI's pattern is "correct" but wrong for *this* codebase.

#### Type 3: Dependency Drift

The spec lists approved dependencies, but the AI introduces new ones.

```markdown
# Spec says:
"Date handling SHALL use date-fns."

# AI generates:
import dayjs from 'dayjs';
// Introduces dayjs instead of using date-fns
```

Now you have two date libraries in your bundle. The AI chose dayjs because it "knows" dayjs from its training data. It doesn't know about your team's decision to standardize on date-fns.

> **Professor's Aside:** I want to be clear about something. The AI isn't being malicious or lazy when it drifts. It's doing its best based on the patterns it's seen in training. The problem is that *your* specific patterns and decisions aren't in its training data. This is why we need automated enforcement -- because educating the AI through prompts is unreliable, but catching drift through linters is deterministic.

## 2.2 How AI Companies Think About Intent Drift

### OpenAI's RLHF: Preventing Drift at the Model Level

OpenAI's Reinforcement Learning from Human Feedback (RLHF) is fundamentally a system for preventing intent drift -- but at the model level rather than the code level.

The process works like this:

1. **The "spec" is human preferences.** Human evaluators define what "good" and "bad" responses look like.
2. **The model generates responses.** These are the "implementations."
3. **A reward model evaluates alignment.** This is the "linter" -- it checks whether the response matches human preferences.
4. **The model is fine-tuned.** Responses that drift from human preferences are penalized.

The key insight: **RLHF is an automated linting system for behavior.** It continuously checks model outputs against a specification (human preferences) and corrects drift.

### Anthropic's RLAIF and Constitutional AI: Self-Linting

Anthropic takes this further with RLAIF (Reinforcement Learning from AI Feedback) and Constitutional AI:

1. **The "spec" is a set of principles** (the constitution) -- explicit rules about how Claude should behave.
2. **The model generates responses** and then *evaluates its own responses* against the principles.
3. **Responses that violate principles are revised** -- the model essentially "lints" itself.
4. **The revised responses are used for training** -- creating a feedback loop.

This is remarkably similar to what we're going to build for code: a system where the spec is encoded as rules, the AI's output is checked against those rules, and violations are flagged automatically.

### Google's Gemini: Structured Output Validation

Google's approach with Gemini emphasizes structured outputs -- constraining the AI to produce output in a specific format that can be validated against a schema. Their Gemini Code Assist tool uses project-level configurations to define coding standards, and the AI's output is validated against these configurations before being presented to the developer.

This is analogous to the `.cursorrules` pattern we'll discuss later in this chapter: persistent configuration files that constrain the AI's behavior.

## 2.3 Custom ESLint Rules as Spec Enforcers

### ESLint: Your First Line of Automated Defense

ESLint (and its ecosystem) is the most practical tool for enforcing spec constraints in JavaScript/TypeScript projects. Let's build some custom rules.

### Example 1: Enforcing Tailwind-Only Styling

Remember our spec constraint: "Use Tailwind CSS for all styling. No custom CSS files."

```javascript
// eslint-rules/no-custom-css-imports.js

/**
 * ESLint rule: no-custom-css-imports
 *
 * Enforces spec constraint: "All styling SHALL use Tailwind CSS.
 * No custom CSS files shall be created or imported."
 *
 * @spec styling-standards v2.1
 * @specLine 3
 */
module.exports = {
  meta: {
    type: 'problem',
    docs: {
      description: 'Disallow importing custom CSS files (spec: Tailwind-only styling)',
      category: 'Spec Compliance',
    },
    messages: {
      noCustomCSS:
        'Custom CSS imports are not allowed. Use Tailwind utility classes instead. ' +
        '(Spec: styling-standards v2.1, line 3)',
    },
    schema: [],
  },

  create(context) {
    return {
      ImportDeclaration(node) {
        const source = node.source.value;

        // Allow Tailwind's base imports
        if (source.includes('tailwindcss') || source === './globals.css') {
          return;
        }

        // Flag any .css, .scss, .less, or .styled imports
        if (/\.(css|scss|sass|less|styled)$/.test(source)) {
          context.report({
            node,
            messageId: 'noCustomCSS',
          });
        }
      },

      // Also catch CSS-in-JS patterns if spec prohibits them
      TaggedTemplateExpression(node) {
        if (
          node.tag.type === 'Identifier' &&
          node.tag.name === 'css'
        ) {
          context.report({
            node,
            messageId: 'noCustomCSS',
          });
        }
      },
    };
  },
};
```

### Example 2: Enforcing State Management Patterns

```javascript
// eslint-rules/enforce-zustand-state.js

/**
 * ESLint rule: enforce-zustand-state
 *
 * Enforces spec constraint: "All shared state SHALL be managed through
 * Zustand stores. Local React state (useState) is only permitted for
 * UI-only state that doesn't leave the component."
 *
 * @spec state-management v1.0
 * @specLine 1-2
 */
module.exports = {
  meta: {
    type: 'suggestion',
    docs: {
      description: 'Enforce Zustand for shared state management',
      category: 'Spec Compliance',
    },
    messages: {
      preferZustand:
        'Consider using a Zustand store instead of useState for state that may be ' +
        'shared across components. (Spec: state-management v1.0)',
      noRedux:
        'Redux is not permitted. Use Zustand for state management. ' +
        '(Spec: state-management v1.0, line 1)',
      noMobx:
        'MobX is not permitted. Use Zustand for state management. ' +
        '(Spec: state-management v1.0, line 1)',
    },
    schema: [],
  },

  create(context) {
    return {
      // Flag imports of prohibited state libraries
      ImportDeclaration(node) {
        const source = node.source.value;

        if (source === 'redux' || source.startsWith('@reduxjs/')) {
          context.report({ node, messageId: 'noRedux' });
        }

        if (source === 'mobx' || source.startsWith('mobx-')) {
          context.report({ node, messageId: 'noMobx' });
        }
      },

      // Warn on useState usage in files that look like they manage shared state
      CallExpression(node) {
        if (
          node.callee.type === 'Identifier' &&
          node.callee.name === 'useState'
        ) {
          const filename = context.getFilename();

          // If the file is in a 'stores/' or 'state/' directory,
          // useState is definitely wrong
          if (filename.includes('/stores/') || filename.includes('/state/')) {
            context.report({ node, messageId: 'preferZustand' });
          }
        }
      },
    };
  },
};
```

### Example 3: Enforcing Error Handling Patterns

```javascript
// eslint-rules/enforce-result-type.js

/**
 * ESLint rule: enforce-result-type
 *
 * Enforces spec constraint: "All service layer functions SHALL return
 * Result<T, E> types instead of throwing exceptions."
 *
 * @spec error-handling v1.2
 * @specLine 4
 */
module.exports = {
  meta: {
    type: 'problem',
    docs: {
      description: 'Enforce Result type pattern for error handling in service layer',
      category: 'Spec Compliance',
    },
    messages: {
      noThrowInService:
        'Service functions must not throw exceptions. Return a Result type instead. ' +
        '(Spec: error-handling v1.2, line 4)',
      noTryCatchInService:
        'Service functions should not use try/catch for flow control. ' +
        'Use Result types for error propagation. ' +
        '(Spec: error-handling v1.2, line 4)',
    },
    schema: [],
  },

  create(context) {
    const filename = context.getFilename();

    // Only enforce in service layer files
    if (!filename.includes('/services/') && !filename.includes('.service.')) {
      return {};
    }

    return {
      ThrowStatement(node) {
        context.report({ node, messageId: 'noThrowInService' });
      },

      TryStatement(node) {
        context.report({ node, messageId: 'noTryCatchInService' });
      },
    };
  },
};
```

### Registering Custom Rules

```javascript
// .eslintrc.js
module.exports = {
  plugins: ['spec-compliance'],
  rules: {
    'spec-compliance/no-custom-css-imports': 'error',
    'spec-compliance/enforce-zustand-state': 'warn',
    'spec-compliance/enforce-result-type': 'error',
  },

  // Override for test files -- different rules apply
  overrides: [
    {
      files: ['**/*.test.ts', '**/*.spec.ts'],
      rules: {
        'spec-compliance/enforce-result-type': 'off',
      },
    },
  ],
};
```

> **Professor's Aside:** Notice how each rule references a specific spec and spec line. This traceability is crucial. When a developer (or an AI) triggers a lint error, they should immediately be able to look up *why* this constraint exists. The lint error isn't just "don't do this" -- it's "the spec says this, for these reasons."

## 2.4 Architecture Decision Records (ADRs) as Formalized Constraints

### What Are ADRs?

Architecture Decision Records are documents that capture important architectural decisions, their context, and their consequences. In SDD, ADRs serve as the *rationale* behind spec constraints.

### ADR Template for SDD

```markdown
# ADR-003: Use Zustand for State Management

## Status
Accepted

## Date
2026-01-15

## Context
We are building a React-based SaaS dashboard that requires shared state across
many components. We need a state management solution that is:
- Lightweight (small bundle impact)
- Type-safe (works well with TypeScript)
- Simple (low learning curve for the team and AI assistants)
- Debuggable (devtools support)

## Decision
We will use Zustand for all shared state management.

## Constraints (to be enforced)
1. All shared state SHALL be managed through Zustand stores
2. React useState is ONLY permitted for component-local UI state
3. No other state management libraries (Redux, MobX, Jotai, Recoil) SHALL be used
4. Zustand stores SHALL be defined in `/src/stores/` directory
5. Each domain SHALL have its own store file (e.g., `user-store.ts`, `cart-store.ts`)

## Consequences
- Positive: Consistent state patterns across the codebase
- Positive: AI assistants can be constrained to generate Zustand-compatible code
- Negative: Some complex state machines may be less elegant than with XState
- Mitigation: For complex state machines, we will use a Zustand middleware pattern

## Enforcement
- ESLint rule: `spec-compliance/enforce-zustand-state`
- CI check: Scan for unauthorized state management imports
- AI constraint: Include in .cursorrules and CLAUDE.md

## Related Specs
- state-management v1.0
- frontend-architecture v2.3
```

### How ADRs Connect to the SDD Pipeline

```
ADR (Why) --> Spec Constraint (What) --> ESLint Rule (How) --> CI Check (When)
```

The ADR explains *why* a decision was made. The spec constraint defines *what* must be done. The ESLint rule implements *how* to check it. The CI pipeline determines *when* it's checked.

This chain is what makes SDD enforceable. Without it, constraints live only in someone's memory or in a document nobody reads.

## 2.5 Persistent Spec Constraints: .cursorrules, CLAUDE.md, and IDE Configuration

### The Problem of AI Amnesia

Every time you start a new conversation with an AI assistant, it has no memory of your previous decisions. It doesn't know about your ADRs. It doesn't know you chose Zustand over Redux. It doesn't know your error handling pattern.

Unless you tell it. Every time.

This is where persistent configuration files come in. They inject spec constraints into the AI's context automatically, so you don't have to repeat yourself.

### .cursorrules (Cursor IDE)

The `.cursorrules` file is read by Cursor IDE and injected into every AI prompt. It acts as a persistent spec constraint:

```markdown
# .cursorrules

## Project: SaaS Dashboard v2

## Architecture
- Framework: Next.js 14 with App Router
- Language: TypeScript (strict mode)
- Styling: Tailwind CSS ONLY. No custom CSS files. No CSS-in-JS.
- State Management: Zustand. No Redux, MobX, or other state libraries.
- Data Fetching: TanStack Query (React Query). No raw fetch in components.
- Forms: React Hook Form with Zod validation schemas.

## Code Patterns
- Error handling: Use Result<T, E> types in service layer. Never throw in services.
- API routes: Use Zod schemas for request validation. Return typed responses.
- Components: Functional components only. No class components.
- Exports: Named exports only. No default exports (except Next.js pages).

## File Structure
- Components: /src/components/{domain}/{ComponentName}.tsx
- Stores: /src/stores/{domain}-store.ts
- Services: /src/services/{domain}.service.ts
- API routes: /src/app/api/{domain}/route.ts
- Types: /src/types/{domain}.types.ts

## Testing
- Framework: Vitest for unit tests, Playwright for E2E
- Every function must have tests before implementation
- Test files: co-located with source ({file}.test.ts)

## Constraints
- NO console.log in production code (use structured logging)
- NO any type (use unknown if type is genuinely uncertain)
- NO synchronous file I/O
- ALL dates use date-fns (no Moment.js, no dayjs, no manual Date formatting)
- ALL IDs are UUIDs (no auto-increment integers)
```

### The Key Insight: Persistent Constraints as Living Specs

These configuration files are not just "AI instructions." They are **living, versioned spec fragments** that travel with the code. They are:

- **Version controlled** -- changes are tracked in git
- **Code-reviewed** -- changes require team approval
- **Consistent** -- every team member (and every AI) gets the same constraints
- **Enforceable** -- they can be validated against the actual codebase

> **Professor's Aside:** One pattern I've seen work extremely well is keeping these files *generated* from the spec. When you update a spec, a script regenerates the `.cursorrules`, `CLAUDE.md`, and other config files. This ensures the persistent constraints are always in sync with the canonical spec documents. We'll explore this more in Module 5.

## 2.6 Techniques to Detect Drift

### Technique 1: Diff Analysis Against Spec Constraints

After the AI generates code, analyze the diff against known constraints:

```python
# tools/drift-detector.py
"""
Drift Detector: Analyzes git diffs against spec constraints.
Runs as a pre-commit hook or CI step.
"""

import re
import subprocess
import sys
import json
from pathlib import Path
from dataclasses import dataclass, field
from typing import Optional


@dataclass
class DriftViolation:
    file: str
    line: int
    rule: str
    message: str
    spec_ref: str
    severity: str  # 'error' or 'warning'


@dataclass
class DriftRule:
    name: str
    spec_ref: str
    severity: str
    pattern: str  # regex pattern to match
    file_glob: str  # which files to check
    message: str
    negate: bool = False  # if True, flag when pattern is NOT found


# Load rules from spec constraints file
def load_rules(rules_path: str = '.spec-lint.json') -> list[DriftRule]:
    """Load drift detection rules from configuration."""
    with open(rules_path) as f:
        data = json.load(f)

    rules = []
    for rule_data in data['rules']:
        rules.append(DriftRule(**rule_data))
    return rules


# Example rules configuration
EXAMPLE_RULES_CONFIG = """
{
  "rules": [
    {
      "name": "no-custom-css",
      "spec_ref": "styling-standards v2.1, line 3",
      "severity": "error",
      "pattern": "import.*\\\\.css['\\\"]",
      "file_glob": "src/**/*.{ts,tsx}",
      "message": "Custom CSS imports are not allowed. Use Tailwind utility classes.",
      "negate": false
    },
    {
      "name": "no-redux",
      "spec_ref": "state-management v1.0, line 1",
      "severity": "error",
      "pattern": "from ['\\\"](@reduxjs|redux)",
      "file_glob": "src/**/*.{ts,tsx}",
      "message": "Redux is not permitted. Use Zustand for state management.",
      "negate": false
    },
    {
      "name": "no-throw-in-services",
      "spec_ref": "error-handling v1.2, line 4",
      "severity": "error",
      "pattern": "throw (new )?\\\\w+",
      "file_glob": "src/services/**/*.ts",
      "message": "Service functions must return Result types, not throw exceptions.",
      "negate": false
    },
    {
      "name": "no-console-log",
      "spec_ref": "logging-standards v1.0, line 2",
      "severity": "warning",
      "pattern": "console\\\\.(log|debug|info|warn|error)",
      "file_glob": "src/**/*.{ts,tsx}",
      "message": "Use the structured logger instead of console methods.",
      "negate": false
    },
    {
      "name": "no-any-type",
      "spec_ref": "typescript-standards v1.0, line 1",
      "severity": "error",
      "pattern": ":\\s*any[\\\\s;,)>]",
      "file_glob": "src/**/*.{ts,tsx}",
      "message": "The 'any' type is not permitted. Use 'unknown' with type guards.",
      "negate": false
    },
    {
      "name": "no-default-export",
      "spec_ref": "module-standards v1.0, line 5",
      "severity": "error",
      "pattern": "export default",
      "file_glob": "src/**/*.{ts,tsx}",
      "message": "Default exports are not allowed. Use named exports.",
      "negate": false
    }
  ]
}
"""


def get_changed_files() -> list[str]:
    """Get list of files changed in the current diff."""
    result = subprocess.run(
        ['git', 'diff', '--cached', '--name-only', '--diff-filter=ACMR'],
        capture_output=True,
        text=True,
    )
    return [f.strip() for f in result.stdout.strip().split('\n') if f.strip()]


def get_diff_lines(filepath: str) -> list[tuple[int, str]]:
    """Get added/modified lines from a specific file's diff."""
    result = subprocess.run(
        ['git', 'diff', '--cached', '-U0', filepath],
        capture_output=True,
        text=True,
    )

    lines = []
    current_line = 0

    for diff_line in result.stdout.split('\n'):
        # Parse hunk headers to track line numbers
        hunk_match = re.match(r'^@@ -\d+(?:,\d+)? \+(\d+)(?:,\d+)? @@', diff_line)
        if hunk_match:
            current_line = int(hunk_match.group(1))
            continue

        # Track added lines (starts with +, not ++)
        if diff_line.startswith('+') and not diff_line.startswith('+++'):
            lines.append((current_line, diff_line[1:]))  # Remove the + prefix
            current_line += 1
        elif not diff_line.startswith('-'):
            current_line += 1

    return lines


def check_file(filepath: str, rules: list[DriftRule]) -> list[DriftViolation]:
    """Check a file's diff against all applicable rules."""
    from fnmatch import fnmatch

    violations = []
    diff_lines = get_diff_lines(filepath)

    for rule in rules:
        # Check if this rule applies to this file
        if not fnmatch(filepath, rule.file_glob):
            continue

        pattern = re.compile(rule.pattern)

        for line_num, line_content in diff_lines:
            match = pattern.search(line_content)

            if rule.negate:
                # Negate mode: flag when pattern is NOT found
                if not match:
                    violations.append(DriftViolation(
                        file=filepath,
                        line=line_num,
                        rule=rule.name,
                        message=rule.message,
                        spec_ref=rule.spec_ref,
                        severity=rule.severity,
                    ))
            else:
                # Normal mode: flag when pattern IS found
                if match:
                    violations.append(DriftViolation(
                        file=filepath,
                        line=line_num,
                        rule=rule.name,
                        message=rule.message,
                        spec_ref=rule.spec_ref,
                        severity=rule.severity,
                    ))

    return violations


def main():
    rules = load_rules()
    changed_files = get_changed_files()
    all_violations: list[DriftViolation] = []

    for filepath in changed_files:
        violations = check_file(filepath, rules)
        all_violations.extend(violations)

    # Report
    errors = [v for v in all_violations if v.severity == 'error']
    warnings = [v for v in all_violations if v.severity == 'warning']

    if warnings:
        print(f"\n{'='*60}")
        print(f"SPEC DRIFT WARNINGS: {len(warnings)}")
        print(f"{'='*60}")
        for v in warnings:
            print(f"  [{v.rule}] {v.file}:{v.line}")
            print(f"    {v.message}")
            print(f"    Spec: {v.spec_ref}")
            print()

    if errors:
        print(f"\n{'='*60}")
        print(f"SPEC DRIFT ERRORS: {len(errors)}")
        print(f"{'='*60}")
        for v in errors:
            print(f"  [{v.rule}] {v.file}:{v.line}")
            print(f"    {v.message}")
            print(f"    Spec: {v.spec_ref}")
            print()

        print(f"\nCommit blocked: {len(errors)} spec compliance error(s) found.")
        print("Fix the errors above or update the spec if the constraint has changed.")
        sys.exit(1)

    if not all_violations:
        print("Spec compliance check passed. No drift detected.")

    sys.exit(0)


if __name__ == '__main__':
    main()
```

### Technique 2: Pattern Matching for Approved Dependencies

```typescript
// tools/dependency-auditor.ts

/**
 * Dependency Auditor
 *
 * Checks that only approved dependencies are used in the project.
 * Approved dependencies are listed in the spec: /specs/approved-deps.md
 */

import { readFileSync } from 'fs';

interface AuditResult {
  approved: string[];
  unapproved: string[];
  specRef: string;
}

function loadApprovedDeps(specPath: string): Set<string> {
  const content = readFileSync(specPath, 'utf-8');
  const deps = new Set<string>();

  // Parse the approved deps spec (assumes a markdown list format)
  const lines = content.split('\n');
  for (const line of lines) {
    const match = line.match(/^[-*]\s+`([^`]+)`/);
    if (match) {
      deps.add(match[1]);
    }
  }

  return deps;
}

function loadInstalledDeps(packageJsonPath: string): string[] {
  const content = readFileSync(packageJsonPath, 'utf-8');
  const pkg = JSON.parse(content);

  return [
    ...Object.keys(pkg.dependencies || {}),
    ...Object.keys(pkg.devDependencies || {}),
  ];
}

function auditDependencies(): AuditResult {
  const approved = loadApprovedDeps('specs/approved-deps.md');
  const installed = loadInstalledDeps('package.json');

  const approvedList: string[] = [];
  const unapprovedList: string[] = [];

  for (const dep of installed) {
    if (approved.has(dep)) {
      approvedList.push(dep);
    } else {
      unapprovedList.push(dep);
    }
  }

  return {
    approved: approvedList,
    unapproved: unapprovedList,
    specRef: 'specs/approved-deps.md',
  };
}

// Run the audit
const result = auditDependencies();

if (result.unapproved.length > 0) {
  console.error('\nUNAPPROVED DEPENDENCIES DETECTED:');
  console.error('================================');
  for (const dep of result.unapproved) {
    console.error(`  - ${dep}`);
  }
  console.error(`\nApproved dependencies are listed in: ${result.specRef}`);
  console.error('To add a new dependency, update the spec and get team approval.');
  process.exit(1);
} else {
  console.log('All dependencies are approved per spec.');
}
```

### Technique 3: AST-Based Import Analysis

For deeper analysis, you can inspect the Abstract Syntax Tree of generated code:

```typescript
// tools/import-analyzer.ts

import * as ts from 'typescript';
import { readFileSync, readdirSync, statSync } from 'fs';
import { join, relative } from 'path';

interface ImportViolation {
  file: string;
  line: number;
  importSource: string;
  rule: string;
  message: string;
}

interface ImportRule {
  name: string;
  // Pattern for files this rule applies to
  filePattern: RegExp;
  // Allowed import patterns (regex)
  allowedImports: RegExp[];
  // Blocked import patterns (regex)
  blockedImports: RegExp[];
  message: string;
}

const IMPORT_RULES: ImportRule[] = [
  {
    name: 'component-isolation',
    filePattern: /src\/components\/.+\.tsx?$/,
    allowedImports: [
      /^react/,
      /^next/,
      /^@\/components/,
      /^@\/hooks/,
      /^@\/types/,
      /^@\/lib/,
      /^@\/stores/,
      /^\.\//,     // relative imports within component directory
      /^\.\.\//,   // parent directory imports
      /^class-variance-authority/,
      /^clsx/,
      /^tailwind-merge/,
    ],
    blockedImports: [
      /^@\/services/,  // Components should not import services directly
      /^@\/app\/api/,  // Components should not import API routes
      /^prisma/,       // Components should never touch the database
      /^@prisma/,
    ],
    message: 'Components must not import directly from services, API routes, or database layers. Use hooks or stores as intermediaries.',
  },
  {
    name: 'service-isolation',
    filePattern: /src\/services\/.+\.ts$/,
    allowedImports: [
      /^@\/types/,
      /^@\/lib/,
      /^@\/config/,
      /^prisma/,
      /^@prisma/,
      /^zod/,
      /^date-fns/,
      /^\.\//,
      /^\.\.\//,
    ],
    blockedImports: [
      /^react/,          // Services must not depend on React
      /^@\/components/,  // Services must not depend on UI
      /^@\/hooks/,       // Services must not depend on hooks
      /^@\/stores/,      // Services must not depend on stores
    ],
    message: 'Service layer must not depend on React, components, hooks, or stores. Services should be framework-agnostic.',
  },
];

function analyzeFile(filePath: string): ImportViolation[] {
  const violations: ImportViolation[] = [];
  const content = readFileSync(filePath, 'utf-8');
  const sourceFile = ts.createSourceFile(
    filePath,
    content,
    ts.ScriptTarget.Latest,
    true,
  );

  const relPath = relative(process.cwd(), filePath);

  for (const rule of IMPORT_RULES) {
    if (!rule.filePattern.test(relPath)) continue;

    ts.forEachChild(sourceFile, (node) => {
      if (ts.isImportDeclaration(node)) {
        const moduleSpecifier = (node.moduleSpecifier as ts.StringLiteral).text;
        const line = sourceFile.getLineAndCharacterOfPosition(node.getStart()).line + 1;

        // Check blocked imports
        for (const blocked of rule.blockedImports) {
          if (blocked.test(moduleSpecifier)) {
            violations.push({
              file: relPath,
              line,
              importSource: moduleSpecifier,
              rule: rule.name,
              message: rule.message,
            });
          }
        }
      }
    });
  }

  return violations;
}

function walkDir(dir: string, filePattern: RegExp): string[] {
  const files: string[] = [];

  for (const entry of readdirSync(dir)) {
    const fullPath = join(dir, entry);
    const stat = statSync(fullPath);

    if (stat.isDirectory() && !entry.startsWith('.') && entry !== 'node_modules') {
      files.push(...walkDir(fullPath, filePattern));
    } else if (filePattern.test(fullPath)) {
      files.push(fullPath);
    }
  }

  return files;
}

// Main analysis
const files = walkDir('src', /\.tsx?$/);
const allViolations: ImportViolation[] = [];

for (const file of files) {
  allViolations.push(...analyzeFile(file));
}

if (allViolations.length > 0) {
  console.error(`\nIMPORT POLICY VIOLATIONS: ${allViolations.length}`);
  console.error('========================================');
  for (const v of allViolations) {
    console.error(`\n  [${v.rule}] ${v.file}:${v.line}`);
    console.error(`    Import: ${v.importSource}`);
    console.error(`    ${v.message}`);
  }
  process.exit(1);
} else {
  console.log('Import policy check passed. No violations found.');
}
```

## 2.7 Building Your Own "Spec Linter"

### A Complete Spec Linter Framework

Let's build a comprehensive spec linting framework that can be customized for any project:

```typescript
// tools/spec-linter/index.ts

import { readFileSync, existsSync } from 'fs';
import { resolve, relative } from 'path';

// =============================================
// Types
// =============================================

interface LintResult {
  passed: boolean;
  violations: Violation[];
  warnings: Violation[];
  checkedFiles: number;
  rulesApplied: number;
}

interface Violation {
  rule: string;
  severity: 'error' | 'warning';
  file: string;
  line?: number;
  message: string;
  specRef: string;
  fix?: string;
}

interface SpecRule {
  name: string;
  specRef: string;
  severity: 'error' | 'warning';
  check: (filePath: string, content: string) => Violation[];
}

// =============================================
// Rule Definitions
// =============================================

function createPatternRule(config: {
  name: string;
  specRef: string;
  severity: 'error' | 'warning';
  filePattern: RegExp;
  searchPattern: RegExp;
  message: string;
  fix?: string;
}): SpecRule {
  return {
    name: config.name,
    specRef: config.specRef,
    severity: config.severity,
    check(filePath: string, content: string): Violation[] {
      if (!config.filePattern.test(filePath)) return [];

      const violations: Violation[] = [];
      const lines = content.split('\n');

      for (let i = 0; i < lines.length; i++) {
        if (config.searchPattern.test(lines[i])) {
          violations.push({
            rule: config.name,
            severity: config.severity,
            file: filePath,
            line: i + 1,
            message: config.message,
            specRef: config.specRef,
            fix: config.fix,
          });
        }
      }

      return violations;
    },
  };
}

// =============================================
// Standard Rule Library
// =============================================

const STANDARD_RULES: SpecRule[] = [
  // ---- Styling Constraints ----
  createPatternRule({
    name: 'no-inline-styles',
    specRef: 'styling-standards v2.1',
    severity: 'error',
    filePattern: /\.(tsx|jsx)$/,
    searchPattern: /style\s*=\s*\{\{/,
    message: 'Inline styles are not permitted. Use Tailwind utility classes.',
    fix: 'Replace inline styles with equivalent Tailwind classes.',
  }),

  createPatternRule({
    name: 'no-css-imports',
    specRef: 'styling-standards v2.1',
    severity: 'error',
    filePattern: /\.(ts|tsx|js|jsx)$/,
    searchPattern: /import\s+.*['"].*\.(css|scss|sass|less)['"]/,
    message: 'CSS file imports are not allowed. Use Tailwind utility classes.',
  }),

  // ---- Type Safety Constraints ----
  createPatternRule({
    name: 'no-any-type',
    specRef: 'typescript-standards v1.0',
    severity: 'error',
    filePattern: /\.(ts|tsx)$/,
    searchPattern: /:\s*any\b/,
    message: 'The "any" type is not permitted. Use "unknown" with type guards.',
    fix: 'Replace "any" with "unknown" and add appropriate type guards.',
  }),

  createPatternRule({
    name: 'no-ts-ignore',
    specRef: 'typescript-standards v1.0',
    severity: 'error',
    filePattern: /\.(ts|tsx)$/,
    searchPattern: /@ts-ignore/,
    message: '@ts-ignore is not permitted. Fix the type error or use @ts-expect-error with a reason.',
    fix: 'Replace @ts-ignore with @ts-expect-error and add explanation.',
  }),

  // ---- Error Handling Constraints ----
  createPatternRule({
    name: 'no-throw-in-services',
    specRef: 'error-handling v1.2',
    severity: 'error',
    filePattern: /src\/services\/.*\.ts$/,
    searchPattern: /\bthrow\s+/,
    message: 'Service functions must not throw. Return Result<T, E> types instead.',
  }),

  // ---- Logging Constraints ----
  createPatternRule({
    name: 'no-console-log',
    specRef: 'logging-standards v1.0',
    severity: 'warning',
    filePattern: /src\/(?!.*\.(test|spec)\.).*\.(ts|tsx)$/,
    searchPattern: /console\.(log|debug|info|warn|error)\(/,
    message: 'Use the structured logger instead of console methods.',
    fix: 'Replace console.xxx() with logger.xxx() from @/lib/logger.',
  }),

  // ---- Security Constraints ----
  createPatternRule({
    name: 'no-hardcoded-secrets',
    specRef: 'security-standards v1.0',
    severity: 'error',
    filePattern: /\.(ts|tsx|js|jsx)$/,
    searchPattern: /(password|secret|api_key|apikey|token)\s*[:=]\s*['"][^'"]+['"]/i,
    message: 'Potential hardcoded secret detected. Use environment variables.',
    fix: 'Move secrets to .env and access via process.env.VARIABLE_NAME.',
  }),

  // ---- Naming Convention Constraints ----
  {
    name: 'component-naming',
    specRef: 'naming-conventions v1.0',
    severity: 'warning',
    check(filePath: string, content: string): Violation[] {
      if (!/src\/components\/.*\.tsx$/.test(filePath)) return [];

      const violations: Violation[] = [];
      const fileName = filePath.split('/').pop()!;

        violations.push({
          rule: 'component-naming',

          message: `Component file "${fileName}" should be PascalCase (e.g., UserProfile.tsx).`,
          specRef: 'naming-conventions v1.0',
        });
      }

    },
  },

  // ---- Architecture Layer Constraints ----
  {
    name: 'layer-isolation',
    specRef: 'architecture v2.3',
    severity: 'error',
    check(filePath: string, content: string): Violation[] {
      const violations: Violation[] = [];

      // Components should not import from services directly
      if (/src\/components\//.test(filePath)) {
        const lines = content.split('\n');
        for (let i = 0; i < lines.length; i++) {
          if (/from\s+['"]@\/services\//.test(lines[i])) {
            violations.push({
              rule: 'layer-isolation',
              severity: 'error',
              file: filePath,
              line: i + 1,
              message: 'Components must not import from the service layer directly. Use hooks or stores as intermediaries.',
              specRef: 'architecture v2.3, layer isolation principle',
            });
          }
        }
      }

      return violations;
    },
  },
];

// =============================================
// Linter Engine
// =============================================

class SpecLinter {
  private rules: SpecRule[] = [];

  constructor(rules: SpecRule[] = STANDARD_RULES) {
    this.rules = rules;
  }

  addRule(rule: SpecRule): void {
    this.rules.push(rule);
  }

  lintFile(filePath: string): Violation[] {
    if (!existsSync(filePath)) return [];

    const content = readFileSync(filePath, 'utf-8');
    const violations: Violation[] = [];

    for (const rule of this.rules) {
      violations.push(...rule.check(filePath, content));
    }

    return violations;
  }

  lintFiles(filePaths: string[]): LintResult {
    const allViolations: Violation[] = [];

    for (const filePath of filePaths) {
      allViolations.push(...this.lintFile(filePath));
    }

    const errors = allViolations.filter(v => v.severity === 'error');
    const warnings = allViolations.filter(v => v.severity === 'warning');

    return {
      passed: errors.length === 0,
      violations: errors,
      warnings,
      checkedFiles: filePaths.length,
      rulesApplied: this.rules.length,
    };
  }
}

export { SpecLinter, SpecRule, Violation, LintResult, STANDARD_RULES };
```

## 2.9 Real-World Examples of Drift

### Case Study 1: The Gradual Migration That Never Happened

**The spec:** "All new components SHALL use the new design system (v2). Legacy components will be migrated on a rolling basis."

**What happened:** An AI assistant was used to build new features. The AI was given the spec, and initially it generated components using the v2 design system. But the existing codebase was full of v1 patterns. Over time, as the AI referenced existing code for context, it started mixing v1 and v2 patterns. After three months:

- 40% of new components used v2 exclusively
- 35% used a mix of v1 and v2
- 25% used v1 exclusively (the AI copied existing patterns)

**How it could have been caught:**

```javascript
// ESLint rule: enforce-design-system-v2
// Flags any import from the v1 design system in new files
createPatternRule({
  name: 'no-design-system-v1',
  specRef: 'design-system-migration v1.0',
  severity: 'error',
  filePattern: /src\/components\/.*\.tsx$/,
  searchPattern: /from\s+['"]@\/design-system\/v1/,
  message: 'New components must use design system v2. Import from @/design-system/v2 instead.',
});
```

### Case Study 2: The Unauthorized Database Migration

**The spec:** "The application SHALL use PostgreSQL with Prisma ORM. All database access SHALL go through the service layer."

**What happened:** A developer asked the AI to "add a quick caching layer for user lookups." The AI generated a solution using Redis -- a perfectly reasonable engineering choice, but one that introduced:

- A new runtime dependency (Redis)
- A new infrastructure requirement (Redis server)
- Direct database access from the cache layer (bypassing the service layer)
- A different data access pattern than the rest of the application

The code worked perfectly in development (the AI helpfully included Redis connection setup). It only failed in the staging environment, which didn't have a Redis instance.

**How it could have been caught:**

```typescript
// Dependency audit would have flagged 'ioredis' or 'redis' as unapproved
// Import analyzer would have flagged direct database access outside service layer
// Architecture layer check would have flagged the violation

// Additionally, a spec linter rule for infrastructure dependencies:
createPatternRule({
  name: 'approved-infrastructure-only',
  specRef: 'infrastructure v2.0',
  severity: 'error',
  filePattern: /\.(ts|tsx)$/,
  searchPattern: /from\s+['"](?:ioredis|redis|mongodb|mongoose|dynamodb)/,
  message: 'Only approved infrastructure dependencies may be used. Current approved: PostgreSQL (via Prisma). See specs/infrastructure.md.',
});
```

### Case Study 3: The Silent Type Erosion

**The spec:** "TypeScript strict mode SHALL be enforced. No use of `any` type."

**What happened:** Over 200 AI-assisted commits, the use of `any` crept in gradually:

1. First, `any` appeared in test files (seemed harmless)
2. Then in utility functions ("it's just a helper")
3. Then in type definitions (`Record<string, any>`)
4. Then in service layer functions
5. Finally in API route handlers

By the time someone noticed, there were 147 instances of `any` in the codebase. The TypeScript strict mode setting was still enabled, but `any` effectively bypasses it.

**How it could have been caught:**

```yaml
# In the CI pipeline: zero-tolerance policy for `any`
- name: Type Safety Audit
  run: |
    ANY_COUNT=$(grep -r ": any\b" src/ --include="*.ts" --include="*.tsx" -c || echo 0)
    if [ "$ANY_COUNT" -gt "0" ]; then
      echo "ERROR: Found $ANY_COUNT instances of 'any' type in source code."
      echo "Spec violation: typescript-standards v1.0"
      grep -rn ": any\b" src/ --include="*.ts" --include="*.tsx"
      exit 1
    fi
```

> **Professor's Aside:** This last case is my favorite because it illustrates the boiling frog problem. No single instance of `any` was alarming. Each one had a plausible justification. But cumulatively, they eroded the type safety that the spec was designed to guarantee. This is exactly why automated enforcement matters -- humans rationalize exceptions, but linters don't.

```
  | Spec       |  | AI Context   |
  | Add Rule   |  | (.cursorrules|
  +-----+------+  | CLAUDE.md)   |
        |         +-----+--------+
        v               |
  +-----+------+        |
  | Add Test   |        |
  | for Gap    |<-------+
  +------------+
```

The critical insight is that every drift violation should feed back into the system in one of two ways:

1. **The spec had a gap** -- update the spec and add a new rule to prevent future drift
2. **The AI drifted from a clear spec** -- update the AI context files to reinforce the constraint

## Summary

In this chapter, we've explored the concept of intent drift and built a comprehensive system for detecting and preventing it.

**Key takeaways:**

1. **Intent drift is the silent killer of AI-assisted development.** Code can be functionally correct while violating architectural decisions, style constraints, and dependency policies.

2. **Three types of drift:** Architectural (wrong technology), Pattern (wrong approach), and Dependency (wrong library). Each requires different detection strategies.

3. **ESLint rules are your first line of defense.** Custom rules that reference spec constraints create a deterministic enforcement layer.

4. **Persistent configuration files (.cursorrules, CLAUDE.md) reduce drift** by injecting constraints into the AI's context automatically.

5. **CI/CD integration makes enforcement automatic.** Spec compliance checks should run on every commit and every PR, just like tests.

6. **The feedback loop matters.** Every drift violation should result in either a spec update or an AI context update, preventing the same drift from recurring.

7. **RLHF and RLAIF are industry-scale drift prevention** -- the same principles that Anthropic and OpenAI use to align their models are the principles you should use to align your AI-generated code with your specs.
