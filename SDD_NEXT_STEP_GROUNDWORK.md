# SDD Next Step: Specs as a Type System — Groundwork

> **"Discipline can drift. Infrastructure cannot."**

---

## 1. The Theoretical Foundation Already Exists

The sddbook already contains the pieces — they just haven't been assembled into a single system yet:

- **MODULE_05 CH01** teaches the **Reverse Spec Technique**: write a spec for what EXISTS, then compare it to a Target Spec. That's the code-to-spec direction.
- **MODULE_03 CH02** builds custom lint rules as spec enforcers — a primitive spec type checker.
- **MODULE_03 CH03** introduces the **Registry Pattern** — a master index of specs with dependency metadata. That's the foundation of a dependency graph.
- **MODULE_04 CH02** gives specs semantic versioning with breaking/additive/patch semantics — type compatibility rules.

The analogy table below isn't a metaphor. It's a **design document**.

---

## 2. The Spec Type System Analogy

| Programming Concept | Spec Equivalent | What It Catches |
|---|---|---|
| Type definition | Constraint — defines what's allowed | Malformed specs, invalid constraint values |
| Function signature | AC — defines input to output | Missing outputs, undefined error cases |
| Import statement | `depends_on` — creates a contract between specs | Broken references, version mismatches |
| Type error | Spec conflict — caught before tests run | Two specs that contradict each other |
| Unused variable | Orphan constraint — spec logic that no AC references | Dead logic or missing test coverage |
| Missing null check | Spec gap — a path with no AC coverage | Untested edge cases, unhandled inputs |

Right now, specs are **untyped documents**. They're YAML files that humans and AI read and interpret. There's no compiler. The only "type checker" is someone reading two specs and noticing they disagree.

Building that type system — the dependency graph, the conflict linter, the path coverage matrix — is what turns SDD from a discipline into an **infrastructure**.

## 3. Part 1: Code to Spec (The Reverse Compiler)

Given an existing codebase without specs, the reverse compiler extracts structured spec drafts.

### 3.1 Structural Extraction

AST parsing to extract function signatures, types, imports, exports, route definitions, and schema definitions.

- **TypeScript**: `ts-morph` or the TypeScript compiler API
- **Python**: `ast` module, `libcst` for concrete syntax trees
- **Database**: Prisma schema, SQLAlchemy models, Alembic migrations
- **Validation**: Zod schemas, Pydantic models, JSON Schema definitions

Output: a skeleton of every module's public interface — what it accepts, what it returns, what it imports.

### 3.2 Behavioral Inference from Tests

Existing tests contain implicit acceptance criteria. A test that asserts:

```typescript
expect(result.status).toBe(401);
```

implies an AC: *"Returns 401 when the request is unauthorized."*

The reverse compiler parses test files, extracts assertion patterns, and maps them to behavioral statements. This gives you the AC section of the micro-spec, derived from what the code already validates.

### 3.3 Constraint Detection

Constraints hide in code as:

- Zod/Pydantic validation rules (`z.string().min(1).max(255)`)
- TypeScript types and interfaces (required vs. optional fields)
- Validation middleware (auth checks, rate limits)
- Environment variable reads (`process.env.DATABASE_URL`)
- Configuration files (`.env`, `docker-compose.yml`, Terraform)

The reverse compiler extracts these and converts them to constraint entries in the spec.

### 3.4 AI-Assisted Gap Filling

The structural extraction gives you the skeleton. An LLM reads the code + extracted skeleton and drafts the Context/Objective/Constraints in micro-spec format. The output is marked `status: draft` — a human must review before it becomes authoritative.

### 3.5 Output Format

```yaml
# Auto-generated by spec-reverse v0.1.0
# Source: src/payments/create-intent.ts
# Status: DRAFT — requires human review

spec:
  id: payment-create-intent
  version: 1.0.0-draft
  status: draft
  generated_from:
    source_file: src/payments/create-intent.ts
    test_files:
      - tests/payments/create-intent.test.ts
    extraction_date: 2026-03-27

  context:
    system: Payment processing service
    dependencies:
      - stripe-sdk
      - prisma-client
    # AI-inferred from import analysis

  objective: |
    Create a Stripe PaymentIntent for a booking session.
    # AI-inferred from function name + test descriptions

  constraints:
    - amount must be positive integer in cents  # from: z.number().int().positive()
    - currency must be supported ISO 4217 code  # from: z.enum(['usd', 'cad', ...])
    - user must be authenticated                # from: authMiddleware usage

  acceptance_criteria:
    - id: AC-01
      description: Returns PaymentIntent client_secret on success
      # from: test "should return client secret"
    - id: AC-02
      description: Returns 401 when user is not authenticated
      # from: test "should reject unauthenticated requests"
    - id: AC-03
      description: Returns 400 when amount is zero or negative
      # from: z.number().int().positive() — NO MATCHING TEST FOUND
      gap: true
```

Note the `gap: true` flag on AC-03 — the constraint exists in code but no test covers it. The reverse compiler found a **spec gap** before any human read the file.

## 4. Part 2: The Spec Compiler Toolchain

### 4.1 `spec-parse` — YAML to Typed AST

Every spec becomes a structured object with typed fields. Malformed specs fail here.

**Responsibilities:**
- Validate YAML structure against the spec JSON Schema
- Parse constraint expressions into typed objects
- Validate AC signatures (inputs, outputs, error cases declared)
- Produce a typed Spec AST for downstream tools

**Analogous to:** The lexer/parser phase of a compiler. Catches syntax errors.

### 4.2 `spec-resolve` — Dependency Graph Builder

Builds the dependency graph from `depends_on` fields.

**Responsibilities:**
- Resolve all `depends_on` references against the spec registry
- Detect missing references (dangling imports)
- Detect circular dependencies
- Detect version incompatibilities (Spec A depends on B@^1.0, but B is at 2.0)
- Output: a directed acyclic graph of spec relationships

**Analogous to:** The linker. Catches "undefined reference" and "incompatible library version."

### 4.3 `spec-check` — The Type Checker

The core of the system. Walks the dependency graph and validates semantic consistency.

**Check 1: Conflict Detection**

Two specs that contradict each other across the dependency graph.

```
Spec A (user-registration):
  constraints:
    - email: required

Spec B (guest-checkout):
  acceptance_criteria:
    - AC-01: Process checkout when email is absent
  depends_on:
    - user-registration  # CONFLICT: B assumes email optional, A requires it
```

**Check 2: Orphan Detection**

A constraint that no AC references.

```
Spec: payment-processing
  constraints:
    - currency must be ISO 4217       # Referenced by AC-01, AC-03 ✓
    - amount must be positive integer  # Referenced by AC-02 ✓
    - idempotency_key must be unique   # NOT referenced by any AC ✗ ORPHAN
```

Orphan constraints are warnings: either the constraint is dead logic, or there's a missing AC (which means a missing test).

**Check 3: Gap Detection**

An input path through the spec with no AC covering it.

```
Spec: user-login
  constraints:
    - password: string, min 8 chars
  acceptance_criteria:
    - AC-01: Successful login with valid credentials
    - AC-02: Reject login with wrong password
    # GAP: No AC for password shorter than 8 chars
    # GAP: No AC for empty password
    # GAP: No AC for account locked after N failures
```

**Check 4: Breaking Change Detection**

Compare current spec version against the previous version in git.

```
BREAKING (requires MAJOR bump):
  - Field removed: response.legacy_token
  - Type changed: amount (string → number)
  - Constraint tightened: max_retries (10 → 5)

ADDITIVE (requires MINOR bump):
  - Field added: response.refresh_token (optional, default: null)
  - Constraint relaxed: supported_currencies (added 'eur')

PATCH (no bump required):
  - Description clarified: AC-03 wording updated
```

**Analogous to:** `tsc --noEmit`. Catches semantic errors without producing output.

### 4.4 `spec-coverage` — Usage Analysis

Extends the existing `check-spec-coverage.py` into a full traceability matrix.

```
Spec ID                  │ ACs │ Tests │ Code Files │ Coverage │ Status
─────────────────────────┼─────┼───────┼────────────┼──────────┼────────
payment-create-intent    │  6  │  11   │     2      │  100%    │ ✓ FULL
user-registration        │  8  │   5   │     3      │   62%    │ ✗ GAPS
notification-preferences │  4  │   4   │     1      │  100%    │ ✓ FULL
guest-checkout           │  5  │   0   │     0      │    0%    │ ✗ SPEC ONLY
```

**Analogous to:** "Are all types used?" — detecting dead code and unimplemented interfaces.

### 4.5 `spec-sync` — CI Enforcer

Runs in CI. Fails the build when the spec contract is violated.

**Enforces:**
- Code files must have `@spec` annotations linking to their spec
- Test files must have `@test` annotations tracing to specific ACs
- Spec changes must have corresponding test changes in the same PR
- Breaking spec changes must bump the major version
- New code files must reference an existing spec (no unspecced code)
- Tier-appropriate strictness:

| Check | Tier 1 (Security) | Tier 2 (Business) | Tier 3 (Utility) |
|---|---|---|---|
| Conflict detection | Error | Error | Warning |
| Orphan constraints | Error | Warning | Info |
| Gap detection | Error | Warning | Off |
| Breaking change enforcement | Error + migration spec required | Error | Warning |
| Spec coverage minimum | 100% | 80% | 50% |

## 5. The Build Sequence

### Phase 1: Schema Definition + `spec-parse`

Define the canonical YAML schema for micro-specs as a JSON Schema. This is the "type definition for types" — without it, nothing else works.

**Deliverable:** JSON Schema for `.spec.yaml` files + a parser that validates against it.

### Phase 2: `spec-resolve` + Dependency Graph

Build the `depends_on` resolver. Parse all specs in a project, build the graph, detect structural issues (cycles, missing refs, version conflicts).

**Deliverable:** CLI tool that outputs the spec dependency graph and structural errors.

### Phase 3: Reverse Compiler (Code to Spec)

Use structural extraction + AI gap-filling. Target JWTMS and OpenWatch as test cases — the audits already identify exactly which modules need specs first.

**Deliverable:** CLI tool that generates draft `.spec.yaml` files from existing TypeScript/Python code.

### Phase 4: `spec-check` (Conflict + Orphan Detection)

Start with the two simplest checks: contradictions between connected specs, and constraints with no AC. These catch real bugs with minimal false positives.

**Deliverable:** CLI tool that runs type-checking passes over the spec graph.

### Phase 5: CI Integration (`spec-sync`)

Wire into existing pipelines alongside `check-spec-coverage.py`. GitHub Actions workflow that gates PRs on spec consistency.

**Deliverable:** GitHub Action + configuration for tiered enforcement.

### Phase 6: Gap Detection (AI-Assisted)

The hardest check — requires reasoning about input spaces. An LLM reads a spec and identifies uncovered paths: "You have no AC for the case where `amount` is zero." This is where the tooling itself becomes AI-assisted.

**Deliverable:** AI-powered gap analysis integrated into `spec-check`.

## 6. Honest Caveats

### Semantic Conflict Detection Is Hard

A pure structural checker catches `required: true` vs `optional: true`. It won't catch "Spec A assumes synchronous processing" vs "Spec B assumes eventual consistency." Subtle conflicts require AI-assisted semantic checking, which means the tool itself uses the same LLMs it governs. That's a feature, not a bug — but it means the spec compiler has a **probabilistic layer**, unlike a traditional type checker. The tooling needs to clearly distinguish between structural checks (deterministic, high confidence) and semantic checks (AI-assisted, flagged for human review).

### The Cold-Start Problem

For greenfield projects, specs come first and the system works naturally. For existing codebases (the actual use case with JWTMS and OpenWatch), the reverse compiler must bootstrap — and those draft specs will have errors. The adoption story must be **gradual**: draft specs → human review → promoted to authoritative → enforcement enabled. Not "flip the switch."

### Spec Granularity Is a Design Decision

Too coarse (one spec per service) and the type system can't catch fine-grained conflicts. Too fine (one spec per function) and maintaining the graph becomes the new burden. The tiered approach already handles this intuitively:

- **Tier 1 modules**: Fine-grained specs (per-endpoint, per-function)
- **Tier 2 modules**: Module-level specs with AC per public method
- **Tier 3 modules**: Lightweight specs, structural validation only

The tooling must formalize this.

### The Tooling Paradox

Building spec infrastructure requires specs for the spec tools themselves. This is both a challenge and a proof-of-concept opportunity: dogfood the system by writing specs for `spec-parse`, `spec-resolve`, and `spec-check` before implementing them.

## 7. What This Means for the SDD Ecosystem

The sddbook teaches the theory. The project audits (JWTMS, OpenWatch) prove the need. The 232 specs and 1,954 tests across existing projects prove the pattern works at scale.

What's missing is the **infrastructure layer** that makes the pattern unforgeable.

With this toolchain:

- **AI collaborators cannot sidestep specs** — `spec-sync` blocks the PR
- **Spec conflicts are caught before tests run** — `spec-check` finds contradictions
- **Existing codebases can adopt SDD incrementally** — the reverse compiler bootstraps draft specs
- **Spec quality is measurable** — orphans, gaps, and coverage are quantified
- **Breaking changes are managed** — semantic versioning with automated detection

The sddbook becomes not just a course, but the **documentation for a spec development kit**. SDD moves from a methodology you teach to an **infrastructure you ship**.

---

## 8. Recommended First Step

Define the **canonical micro-spec YAML schema** as a JSON Schema. Every tool in the chain depends on this definition. It answers:

- What fields are required vs. optional?
- What types do constraint values support?
- How are `depends_on` references structured?
- How are ACs identified and cross-referenced?
- How is versioning encoded?
- How are tiers assigned?

This schema is the **type definition for the type system**. Everything else builds on it.
